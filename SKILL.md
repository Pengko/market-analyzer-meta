# 股票深度分析

围绕单只 A 股完成一份偏交易决策的深度分析，而不是泛泛点评。分析必须从市场环境开始，逐层落到板块、个股趋势、分时主力、筹码结构、技术因子和消息面，最后给出下一交易日和后续几日的预期，并判断当天是否适合入手。
## 分析原则

- 先确认标的，再确认市场与代码格式，避免同名股票误判。
- 分析前必须先确认当前中国时间；优先获取中国网络时间，失败时再退回本地时间。
- 先看数据新鲜度，再下结论；核心行情、资金、筹码、题材、公告至少说明时间范围。
- 必须先判断当前属于盘中、午间休盘、收盘后还是开盘前，再切换对应分析口径，不能混用。
- 大盘环境数据是**强制前置模块**，必须在进入个股分析前完成获取。
- **诊断分析模式**：当用户问"为什么暴跌/跌停"而非"明天怎么看"时，加载 `references/concept-blowoff-analysis-framework.md`，使用概念生命周期+主力盈亏估算框架输出诊断报告（详见该文件）。若本地 `index_daily` 缺失或延迟，必须通过行情 API（如腾讯 `qt.gtimg.cn`）补抓上证/深成/创业板至少两个指数的当日数据，禁止在大盘数据完全缺失的状态下直接分析个股。
- 大盘环境中的**两市成交额**必须做**三重对比**：当日成交额 vs 昨日成交额 vs 近期均量（近5日或近10日均量），仅给一个绝对值不足以判断放量/缩量，必须标注环比变化率和相对均量的偏离度。若昨日成交额无法直接获取，允许通过指数成交量 × 平均价格比例进行估算，但必须在报告中标注`估算`。
- 事实与推断分开写；无法验证的内容明确标注为假设或待核实。
- 必须先分析大盘，再分析板块，再分析个股，不能跳步。
- 分析个股时，默认至少找出同板块里最相关的几只对标个股做联动观察，不能只看目标股单独走势。
- 必须区分交易驱动、题材驱动、资金驱动和基本面驱动。
- 输出必须有结论，但不能掩盖不确定性；必须写明多空判断依据。
- 输出时必须明确标注时间来源，以及每类关键数据来自本地还是网络渠道。
- 输出默认必须使用本技能规定的固定曝光格式；除非用户明确要求别的结构，否则不能自由发挥到把关键模块写丢。
- `对标股联动` 是强制模块，不是可选模块；若缺失该模块，视为本技能输出不合格。
- 若对标股数据缺失，必须单独写出“对标股数据缺失/降级说明”，不能直接省略对标股部分。
- 当任务需覌"抓新闻、补正文、复用登录态、归一化消息"时，通过 `trendradar-mcp` 获取原始新闻数据，由本技能内置的 `_trendradar_to_news_sentiment()` 进行归一化处理。当 TrendRadar 两阶段筛选均为 0 条时，自动 fallback 到 `fetch_browser_news.py`（market-news-intelligence 技能的抓取脚本）补充抓取。
- 若 `portfolio.yaml` 中该股票有持仓（`hold > 0`），自动按 `持有中` 视角分析，无需用户主动提供
- 若用户没有主动提供成本价、仓位、已持有状态，且 `portfolio.yaml` 中也无该股票持仓，默认按 `未持有` 视角分析，不要擅自切到持仓去留模式
- 当运行环境是 `OpenClaw(决策) + Hermes(执行)` 时，浏览器抓取默认走 Hermes 执行层，不直接在 OpenClaw 本地进程里跑浏览器抓取链路。

- 如果某一步失败或降级，必须补一行中文说明，例如：

  - `【消息面】未命中有效结果，先按缺失降级`
  - `【板块判断】已从旧行业口径纠偏到最新题材口径`

#### 脚本执行失败时的固定处理

当某个脚本运行返回非零退出码或空输出时（如 `discover_ths_mobile_stock_concepts.py` exit 1、`get_quote_tencent.py` 空输出）：
1. 第一时间在报告中写出 `脚本名称 执行失败（exit code / 空输出），已降级处理`
2. 立即尝试该数据维度的替代获取方式（如直接 API、本地缓存、浏览器爬取）
3. 若所有替代方式均失败，才能将该维度标记为 `missing`
4. 禁止在没有尝试替代方案之前，直接将整体分析降级或跳过该维度

#### quick_analyze.py / pandas 缺失时的降级路径（2026-05-26 实战验证）

当 `quick_analyze.py` 因环境缺少 `pandas` 而失败（`ModuleNotFoundError: No module named 'pandas'`）时，禁止因此放弃分析。采用以下手动数据获取流程：

1. **实时行情**：直接调用腾讯 API `curl -s "http://qt.gtimg.cn/q=sz/sh{ticker}" | iconv -f gb2312 -t utf-8`
   - 字段解析：逗号分隔，字段3=最新价、字段4=昨收、字段5=开盘、字段6=最高、字段7=最低、字段32=涨跌幅(%)、字段37=成交额(万元)、字段38=换手率(%)
2. **本地日线/因子/资金流向**：使用 `pyarrow.parquet` 直接读取 parquet
   - `pq.read_table('{STOCK_DATA_ROOT}/daily/{code}.parquet').to_pandas()`
   - `pq.read_table('{STOCK_DATA_ROOT}/stk_factor_pro/{code}.parquet').to_pandas()`
   - `pq.read_table('{STOCK_DATA_ROOT}/moneyflow_data/individual/ths/{code}.parquet').to_pandas()`
   - ⚠️ **先检查实际列名**：`stk_factor_pro` 中 KDJ J 值列名为 `kdj_bfq` 而非 `kdj_j_bfq`；`moneyflow` 日期列名为 `trade_date` 而非 `date`。详见 `references/parquet-column-pitfalls.md`
3. **本地筹码**：读取 `cyq_perf/{code}.parquet`（cyq_chips 已废弃）
4. **大盘指数**：腾讯 API `curl -s "http://qt.gtimg.cn/q=sh000001,sz399001,sz399006"`

**关键原则**：
- pyarrow 通常是系统预装的（macOS 常自带），比 pandas 更可能可用
- 若 pyarrow 也不可用，降级至 `python3 -c "import pyarrow.parquet as pq"` 先验证
- 所有降级操作必须在报告中标注 `因 quick_analyze.py 依赖缺失，已降级为直接 API + pyarrow 读取`

   - 执行前必须先检查本地 `index_daily` 最新日期是否覆盖分析日

   - 均线、趋势、动量、波动率、超买超卖
8. 再补资金、筹码、消息面
   - 资金承接、筹码位置、公告与催化
9. 最后给交易结论
   - 当前判断、剩余时段或次日预期、风险位、失效条件

- 必须明确写出缺失项

最新交易日判定规则：

- 第一步：先看中国网络时间，确认当前自然日和盘前 / 盘中 / 盘后
- 第二步：再看浏览器能否确认该股 `当日分时 / 当日最新价 / 当日日K / 当日竞价`
- 第三步：再看本地数据是否已经同步到这个交易日

执行口径：

- 如果浏览器已经能确认当天交易事实，则优先把这一天视为当前有效交易日
- 如果本地数据日期也追平到同一天，则记为 `最新数据已同步`
- 如果浏览器已确认当天事实，但本地数据仍停在前一日，则记为 `当天事实已确认，但本地未同步完成`
- 不能因为某只股票本地日线仍停在前一日，就否定浏览器已经确认的当天交易事实
- 也不能只因为本地数据日期较新，就跳过浏览器核实

补充说明：

- 本地日线通常是收盘后连续更新，更多用于历史结构和同步确认
- 当天是否属于 `最新交易日待验证样本`，默认先由浏览器确认

## 工作流

### 三层架构

1. `dimension analyzers` — 各维度独立出结果（大盘、板块、消息面、对标股联动、个股结构、分时/竞价、资金/筹码/两融/技术因子）
2. `context propagation` — 规则链引擎（`decision/context_propagation_rules.py`）
   - 四层传导：市场→板块→个股→分时→消息面
   - 传递的是“约束和背景”，不直接覆盖原始判断
   - 支持冲突检测与降级逻辑
3. `final decision` — 统一裁决，明确各维度支持/反对理由、冲突项及最终结论成立依据

执行原则：先独立分析，再上下文传递，最后统一裁决。

### 规则链引擎

四层规则链累加 `bias_delta`，确定最终 `action_bias`：

| 字段 | 说明 |
|------|------|
| `action_bias` | supportive / neutral / conservative / defensive |
| `support_flags` | 支持做多标志列表 |
| `risk_flags` | 风险标志列表 |
| `constraint_score` | 约束评分（-10 到 +10） |
| `rule_details` | 每条规则执行结果详情 |

行动偏向判定：overall_bias ≥ 3 → supportive；≤ -3 → defensive；-3~ -1 → conservative；其他 → neutral。

冲突检测自动识别：市场强但板块弱、板块强但个股弱、消息面强但环境弱、竞价积极但分时承接弱。冲突标志自动追加到 `risk_flags`。

消息面规则组：新催化+高可信度（bias_delta +1）；旧消息重炒/消息面缺失（bias_delta 0）。

`build_context_from_payload(payload)` 将 JSON payload 转换为规则引擎上下文。

### build_stock_report.py 超时处理

超时（60秒）时：在报告中标注超时；尝试腾讯API获取行情、本地 `minute_kline.csv`、浏览器搜索板块信息；结合本地数据直接分析，明确标注数据来源。

### Step 1: 确认当前中国时间

- 第一优先级：中国网络时间；不可得则退回本地时间
- 必须在报告中明确写明时间来源

时段划分：盘前（<09:30）、上午盘中（09:30-11:30）、午间休盘（11:30-13:00）、下午盘中（13:00-15:00）、盘后（>15:00）

## 时段分支规则

## 盘前

重点回答"今天大盘和板块大概率会怎么开、这只股今天值得不值得重点盯"。

- 允许重点分析：
  - 前一交易日市场与板块结构
  - 个股月/周/日趋势
  - 融资融券、筹码、技术因子、消息面
- 不要假装已有当日盘中和收盘数据
- 不要分析当日 `rt_min` 实时分时
- 不要分析当日 `stk_auction_c`

**盘前分析日期归属规范**：

### Step 12: 做板块对标联动

  2. 使用腾讯行情 API 批量获取对标股实时数据
  3. 如仍无法获取，必须在报告中写出 `对标股数据缺失/降级说明`
- 优先关注：
  - 板块龙头
  - 逻辑最硬的核心票
  - 最先启动的先手票
  - 同阶段涨幅最大的强势票
- 如果目标股本身不是龙头，要明确它是跟风、补涨、中军，还是分支独立逻辑

联动分析至少要回答：

- 目标股和板块龙头走势是否同步
- 龙头强、目标股弱，说明什么
- 龙头弱、目标股独强，说明什么
- 板块是否整体共振，还是仅少数个股抱团
- 目标股更像受板块带动，还是自身有独立催化

- 最相关的 3-5 只板块对标个股

- 它们的今日涨跌幅；若拿不到今日涨跌幅，则至少列最近一个完整交易日涨跌幅，并明确这是降级结果


必须先看大周期，再看中周期，再看短周期：

  - 今天收盘结构强不强

### Step 6: 用实时分钟数据分析当日主力意图

使用浏览器获取的当日分时走势分析主力意图，重点不是描述走势，而是推断主力行为。

- 必须重点拆解 4 个关键时窗：
  - `09:30-09:35` 开盘初段
  - `09:48-09:56` 首次强冲与承接窗口
  - `11:25-11:30` 上午收盘前 5 分钟
  - `13:00-13:30` 午后开盘前半小时
  - `14:55-15:00` 尾盘最后 5 分钟
- 关注：
  - 开盘是否抢筹
  - 盘中是否主动承接
  - 冲高后是换手还是出货
  - 尾盘是回流抢筹还是资金兑现
  - 关键时段量能是否异常放大
- **涨停炸板专项判断**（冲板后未封住回落）：
  - 核心指标：冲板时段累计成交量 vs 炸板后回落时段累计成交量
  - 若 **冲板量能 < 回落量能**（如 7.9万手 vs 36.5万手），说明板上抛压远大于封板买盘，主力封板意愿不足，优先判定为
- 对每个关键时窗都要判断：
  - 方向：净流入还是净流出
  - 强度：普通波动还是异常波动
  - 结果：价格是否被带动并维持
  - 性质：抢筹 / 试盘 / 护盘 / 兑现 / 出货
- 目标输出：
  - 今天主力意图更像吸筹、试盘、拉升、洗盘、派发还是护盘
  - 4 个关键时窗分别是加分、中性还是减分

关键时窗解读规则：

- `09:30-09:35` 强：只代表初始定价偏强，不代表全天一定强
- `09:48-09:56` 强：若第一次上冲后回撤有限，通常说明主动承接比表面更强
- `11:25-11:30` 强：代表上午走势在午休前被强化
- `13:00-13:30` 强：代表午后资金在半小时内持续认可上午结论，比只看前 5 分钟更可靠
- `14:55-15:00` 强：代表收盘前愿意拿先手，但若全天弱，仅能视为博弈加分

综合判断规则：

- 若 4 个时窗中有 3 个以上偏强，且市场和板块共振，则次日预期显著偏强
- 若只有尾盘最后一个时窗偏强，而前面普遍偏弱，优先视为尾盘博弈而非趋势确认
- 若前强后弱，优先按冲高回落结构处理
- 若前弱后强，优先按修复结构处理
- 若 4 个时窗强弱交错，说明分歧较大，次日预期要降低

午间推演的额外规则：

- 若 `09:48-09:56` 承接强、上午收盘站上均价线、`11:25-11:30` 再次强化，则优先按 `强修复` 处理
- 若只有上午最低点反弹，但首次强冲承接差、午前量能弱，则最多按 `偏弱修复`
- 若上午收盘仍在均价线下，且第一次试盘后明显回落，优先防午后冲高回落

  - `low`

  - 尾盘集合竞价是否明显抢筹
  - 集合竞价成交额是否异常放大
  - 竞价均价 `vwap` 相对收盘价的关系

  - 尾盘资金是偏抢筹、偏中性，还是偏兑现

### Step 7: 先分析融资融券结构

在分析筹码之前，先判断该股是否为融资融券标的，并用融券数据判断空头力量是否在增强。

重点使用：

- `${STOCK_DATA_ROOT}/margin_detail`

分析要点：

- 该股是否存在持续的融资融券数据
- `rzye`、`rqye`、`rzrqye` 的变化方向
- `rzmre`、`rqmcl`、`rzrqjchg` 的阶段变化
- 融资余额增加是助涨还是高位博弈
- 融券压力是否上升，是否对短线预期构成压制

字段解释：

- `rzye`: 融资余额，越高代表融资买盘沉淀越多
- `rqye`: 融券余额，越高代表融券空头头寸越大
- `rzrqye`: 融资融券余额合计
- `rzmre`: 融资买入额，短线增量做多资金的重要观察项
- `rqmcl`: 融券卖出量，短线新增做空压力的重要观察项
- `rzrqjchg`: 融资融券余额差值变化，可用于观察多空净变化方向

解读规则：

- 若 `rzmre` 持续放大且股价走强，通常偏多
- 若 `rqmcl` 明显放大且股价承压，通常偏空
- 若融资余额上升但股价不涨，说明追高筹码可能堆积，需要防兑现
- 若融券余额上升但股价依旧强势，说明承接强，后续可能有挤空效应
- 若融资和融券同时放大，说明分歧扩大，应结合日内分时与收盘结构判断胜负

输出要回答：

- 该股是否可被视为融券股
- 当前融资力量与融券力量谁更强
- 融资融券结构对次日走势是偏多、偏空，还是中性

**分析要点**：
- 获利盘/套牢盘比例
- 加权平均成本 vs 当前价（偏离度）

- 成本集中度（中段50%筹码占价差比）
- 套牢盘压力位（cost_85/cost_95）
- 获利盘支撑位（cost_15/cost_5）
- 成本迁移趋势（近5日均价/获利盘变化）

分析要点：

- 筹码峰主要集中在哪个价格区间
- 当前股价相对主筹码区是低于、接近还是显著高于
- 近期筹码是否上移
- 获利盘和套牢盘分布如何
- 资金净流入是否与价格上涨同步
- 是边拉边派，还是缩量锁仓，还是放量换手
- 先确认筹码、资金、技术因子的最后日期，过期样本只能降权使用

#### 资金数据延迟 1 日时的固定处理

`moneyflow_data/individual` 常常比当日延迟 1 个交易日更新（如 T 日分析时最新为 T-1）。遇到这种情况：
1. 不得将资金维度标记为 `missing`或直接略过，应标记为 `stale_1d`
2. 使用 T-1 日资金数据时，必须在报告中写明 `资金数据延迟 1 日，以 T-1 数据降权参考`
3. 交易结论中，资金维度的权重降为辅助参考，不得用 T-1 日资金流向直接预测 T+1 日走势
4. 优先用分钟线成交量/成交额的日内分布作为 T 日资金意图的替代判断

#### 本地核心字段异常时的交叉验证

若本地 `daily` 或 `moneyflow` 中核心字段（如 `amount`、`vol`、`turnover_rate`）出现明显异常值（如连续多日显示 0 或超低于合理区间）：
1. 必须立即用浏览器/行情 API 的同期数据进行交叉验证
2. 发现矛盾后，以分钟线汇总值或网络渠道为准
3. 在报告中必须写明 `本地数据字段异常，已用 XXX 渠道交叉验证`

#### 本地指数数据延迟时的大盘数据获取

当本地 `index_daily` 数据延迟（如分析日为 T，但本地指数只到 T-1）时，通过行情 API（如腾讯 `qt.gtimg.cn`、同花顺等）补抓当日大盘数据。推荐至少获取上证指数（sh000001）、深证成指（sz399001）、创业板指（sz399006）中两个以上的当日行情。

行情 API 返回格式通常为分号分隔多指数、逗号分隔字段，典型字段含义如下：
- 字段3：最新价
- 字段4：昨收价
- 字段32：涨跌幅（%）
- 字段37：成交额（万元）

示例输出解析：
```
v_sh000001="1~上证指数~000001~4055.55~4027.21~...~28.34~0.70~...~97656549~..."
```
- 最新价：4055.55
- 涨跌：28.34
- 涨跌幅：+0.70%
- 成交额：97656549万元 ≈ 976.57亿元

**报告中必须写明**：`本地指数数据延迟 X 日，当日大盘走势来源为腾讯行情API网络渠道`

#### 通过T日快照反推T-1日大盘收盘数据

当分析日为 T-1（昨天），而当前时间是 T（今天），且本地 `index_daily` 完全没有 T-1 数据时，可通过 T 日腾讯快照的**昨收**字段反推 T-1 收盘价：

- 腾讯快照字段4（昨收）= 上一个交易日的收盘价 = T-1 日收盘
- 结合本地 T-2 日收盘数据，可计算 T-1 涨跌幅：`T-1涨跌幅 = (T-1收盘 - T-2收盘) / T-2收盘`

**示例**：当前 2026-04-24，分析 2026-04-23 大盘。快照显示上证指数昨收=4093.25，即 04-23 收盘=4093.25。本地 04-22 上证收盘=4106.26，则 04-23 涨跌幅 = (4093.25-4106.26)/4106.26 = -0.32%。

**限制**：此方法只能获取 T-1 收盘点位和推算涨跌幅，无法获取 T-1 当日最高/最低/成交量等明细数据。

输出要回答：

- 主力大致成本区
- 主力当前更像锁仓、继续吸筹、拉高试盘、还是准备兑现

#### 筹码结构分析方法（基于 cyq_perf）

使用 `${STOCK_DATA_ROOT}/cyq_perf` 数据进行筹码分析。`cyq_chips` 上游数据质量无效，不再使用。

**数据来源**：`${STOCK_DATA_ROOT}/cyq_perf/{code}.parquet`
**分析函数**：`scripts/analysis/stock_trend_analyzer.py::analyze_chip_structure()`

**cyq_perf 数据字段**：
- `winner_rate`：获利盘比例（%），当前价以下的筹码占比
- `weight_avg`：加权平均成本（元）
- `cost_5pct / cost_15pct / cost_50pct / cost_85pct / cost_95pct`：成本分位数
- `his_low / his_high`：历史最低/最高价

**从 cyq_perf 可推导的分析维度**：
1. **获利盘/套牢盘**：`winner_rate` 直接给出获利盘比例，`100 - winner_rate` = 套牢盘
2. **成本偏离度**：`(close - weight_avg) / weight_avg × 100%`，正值=多数持仓获利
3. **成本集中度**：`(cost_85 - cost_15) / (cost_95 - cost_5)`，越高说明筹码越集中
4. **套牢盘压力位**：`cost_85` 以上为重度套牢区，突破需放量
5. **获利盘支撑位**：`cost_15` 以下为获利盘密集区，回调至此易有承接
6. **成本迁移趋势**：对比近5日 `weight_avg` 和 `winner_rate` 的变化

**报告输出格式**：
```
### 筹码结构分析（数据来源：cyq_perf）

| 指标 | 数值 | 说明 |
|------|------|------|
| 获利盘比例 | XX.X% | 收盘价以下的筹码占比 |
| 套牢盘比例 | XX.X% | 收盘价以上的筹码占比 |
| 加权平均成本 | X.XX元 | 全部筹码加权平均成本 |
| 现价偏离均价 | +X.X% | 当前价高于/低于平均成本 |
| 成本5%分位 | X.XX元 | 最低成本区（强支撑） |
| 成本50%分位 | X.XX元 | 中位成本 |
| 成本95%分位 | X.XX元 | 最高成本区（强压力） |
| 成本集中度 | XX% | 中段50%筹码占价差比 |
```

### Step 9: 量价背离检测

检测价格与技术指标的背离信号，用于判断趋势反转概率。

使用脚本：`/Users/penghongming/agent-skills/custom/stock-deep-analysis/scripts/detect_divergence_v3.py`

背离类型（6种）：

| 类型 | 说明 | 基础命中率 |
|------|------|-----------|
| MACD顶背离 | DIF未随价格创新高 | 56.5% |
| KDJ顶背离 | KDJ高位死叉趋势 | 52.8% |
| RSI顶背离 | RSI未随价格创新高 | 54.5% |
| MACD底背离 | DIF未随价格创新低 | 49.1% |
| KDJ底背离 | KDJ低位金叉趋势 | 46.1% |
| RSI底背离 | RSI未随价格创新低 | 45.5% |

**核心发现（500只股票测试）：**

1. 单纯背离命中率约50%，接近随机
2. **结合趋势位置**可显著提升命中率：

   | 场景组合 | 命中率 | 提升 |
   |---------|--------|------|
   | 底背离 + 震荡_反弹高点 + MACD | **66.0%** | +17% |
   | 底背离 + 震荡_反弹高点 + CCI | 65.4% | +16% |
   | 顶背离 + 上升_上升中继 + KDJ | 62.3% | +10% |
   | 顶背离 + 上升_反弹高点 + CCI | 59.0% | +5% |

3. 关键洞察：
   - **震荡反弹高点 + 底背离**是最强看涨信号场景
   - **上升趋势中的顶背离往往是中继**，不是反转
   - 上升趋势中的底背离是陷阱（37-42%）

**推荐用法：**

```bash
# 增强版背离分析（支持RSI/MACD/KDJ/CCI）
python3 "/Users/penghongming/agent-skills/custom/stock-deep-analysis/scripts/detect_divergence_v3.py" 000001.SZ

# 高置信度筛选
python3 "/Users/penghongming/agent-skills/custom/stock-deep-analysis/scripts/detect_divergence_v3.py" 000001.SZ --min-confidence 0.6

# JSON 输出
python3 "/Users/penghongming/agent-skills/custom/stock-deep-analysis/scripts/detect_divergence_v3.py" 000001.SZ --json
```

**解读规则：**

- 出现底背离时，优先看是否处于**震荡市的反弹高点**，而非上升趋势
- 出现顶背离时，优先看是否处于**上升趋势的中继位置**，而非下降趋势的反弹高点
- 高置信度（>=0.7）信号可靠性更高
- 背离信号需结合趋势位置判断，不能单独作为交易依据

**测试脚本位置：**

- `/Users/penghongming/agent-skills/custom/stock-deep-analysis/scripts/test_divergence_combo_trend.py`

### Step 10: 用专业技术因子做交叉验证

使用 `${STOCK_DATA_ROOT}/stk_factor_pro` 做技术面复核，不要让指标主导结论，只能用于验证。

- 均线是默认必看项，优先检查 `MA5 / MA10 / MA20 / MA30`。
- 分析时默认回答：
  - 当前价在 `MA5 / MA10 / MA20 / MA30` 的上方还是下方
  - 最近的短中期均线是多头发散、粘合还是空头压制
  - 哪条均线最接近当前价，是否可视为第一动态支撑或第一动态压力
- 当个股处于短线交易场景时，默认把均线纳入关键价位判断：
  - `MA5 / MA10` 优先视为短线动态支撑或短线强弱分界
  - `MA20` 视为波段支撑/趋势分界
  - `MA30` 视为更弱情况下的重要防守位
- 输出支撑位和压力位时，不要只写静态前高前低；如果均线位置接近现价，必须把对应均线一并写入。

- 可关注：
  - 趋势类
  - 动量类
  - 波动率类
  - 超买超卖类
- 目标：
  - 验证趋势是否健康
  - 验证是否接近短线过热
  - 验证是否存在背离或衰竭

### Step 11: 收集消息面

如果没有现成 API 或本地消息库，默认必须主动通过浏览器获取消息面，不能因为“没有接口”就把消息面跳过。

收集近几日是否有消息加持，包括但不限于：

- 公司公告
- 政策消息
- 行业催化
- 订单/业绩/重组/回购
- 媒体与市场热议主题

判断标准：

- 消息是实锤还是传闻
- 是旧消息重炒还是新催化
- 对板块还是对个股单独生效
- 持续性强弱

#### 消息获取渠道优先级

默认按下面顺序取消息，优先高可信渠道：

1. `公告 / 官方披露`
   - 巨潮资讯
   - 上交所 / 深交所公告
   - 公司公告
   - 业绩预告 / 业绩快报 / 回购 / 减持 / 重组 / 合同 / 问询回复
2. `政策 / 监管官方渠道`
   - 证监会
   - 工信部
   - 发改委
   - 国务院或地方政府官网
   - 行业协会正式文件
3. `主流财经媒体`
   - 财联社
   - 证券时报
   - 上证报
   - 中证报
4. `行情平台与题材归因渠道`
   - 东方财富
   - 同花顺
   - 开盘啦
   - Yahoo Finance Hong Kong：`https://hk.finance.yahoo.com/`
5. `二手传播 / 社区情绪`
   - 股吧
   - 社媒
   - 群转发
   - 未证实截图

渠道使用规则：

- `第一层` 用来确认事实是否存在
- `第二层` 用来确认政策与监管催化
- `第三层` 用来确认主流叙事是否形成
- `第四层` 用来观察市场如何归因、如何传播、如何情绪化解读
- `第五层` 只能当作情绪噪音参考，不能单独作为结论依据

#### Yahoo Finance Hong Kong 的定位

`https://hk.finance.yahoo.com/` 可以作为消息面补充渠道，但默认不应视为 A 股消息的一手权威源。

适合它的用途：

- 补看快讯流、市场 headlines、海外视角摘要
- 观察市场新闻是否已经扩散成公开叙事
- 辅助判断某条题材是否已经从小范围讨论扩散到更广泛财经媒体

不适合让它单独承担的用途：

- 直接替代交易所公告
- 直接替代公司正式披露
- 单独作为 A 股个股催化是否成立的最终证据

使用时默认结论：

- 若 Yahoo 与公告一致，可作为传播强化证据
- 若 Yahoo 有而公告没有，优先视为媒体层信息，先降级，不直接当实锤
- 若 Yahoo 只是泛市场新闻，对单只 A 股只能作为弱背景，不能直接映射成个股驱动

#### 浏览器获取的执行要求

当分析进入消息面步骤时，默认主动做下面几步：

1. 先查个股公告与交易所披露
2. 再查板块政策或行业催化
3. 再查主流财经快讯
4. 最后查行情平台题材归因与 Yahoo Finance Hong Kong 这类补充渠道

**同花顺 F10 公告页特殊处理（新增）：**
- 同花顺移动端公告详情页（`news.10jqka.com.cn/tapp/notice.html#seq=XXX`）为 Vue SPA，内容通过 JavaScript 动态加载
- curl/urllib 等静态请求只能获取空壳 HTML，无法提取公告正文
- **必须通过浏览器工具**（`browser_navigate` → `browser_snapshot`）才能获取完整内容
- 若浏览器守护进程启动失败，参见 `references/environment-fallback-playbook.md` 第 5 节修复

如果浏览器只拿到了低可信来源：

- 必须明确标注 `消息可信度偏低`
- 必须写出 `暂未找到公告实锤`
- 不能把这类消息写成确定性驱动

默认必须显式补一层 `消息面情绪判断`，不能只罗列新闻标题。

最低合格拆法：

- `消息方向`：偏利多 / 偏利空 / 中性 / 多空交织
- `消息级别`：个股级 / 板块级 / 市场级
- `消息新鲜度`：盘中新增 / 当日新增 / 近 3 日 / 旧消息重炒
- `消息可信度`：公告实锤 / 主流媒体 / 二手转述 / 未证实传闻
- `情绪作用方式`：直接驱动高开、带动板块共振、只提供题材联想、无法解释当前走势

解读要求：

- 不要把“有消息”直接等同于“会涨”
- 真正要判断的是：消息有没有改变资金定价强度
- 若个股强势但没有新增催化，要明确写出“更像纯资金行为”
- 若板块共振明显但个股消息一般，要明确写出“主要受板块情绪带动”
- 若消息已经是旧闻，要优先判断是否属于旧消息重炒，不能按新催化处理
- 若只有传闻没有实锤，默认只能作为弱证据，不能主导结论

输出至少要回答：

- 当前最主要的消息驱动是什么
- 这个驱动偏个股、偏板块还是偏市场
- 这条消息对今天强弱究竟是核心驱动、辅助驱动，还是噪音
- 如果去掉这条消息，走势逻辑是否仍然成立

### Step 13: 形成交易结论

最终必须给出：

- 今天市场环境对这只股是加分还是减分
- 今天板块环境对这只股是加分还是减分
- 个股趋势当前所处阶段
- 主力意图判断
- 次一交易日预期
- 后续 3-5 个交易日预期
- 当天是否适合买入
- 如果不适合买入，应该等什么信号
- 如果可以买，风险位和失效条件是什么
