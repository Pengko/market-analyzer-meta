## 固定曝光格式

除非用户明确要求更短或改格式，否则最终输出默认按下面顺序组织，不能缺段：

1. `场景与数据`
   - 当前属于盘前 / 盘中 / 午间 / 盘后
   - 时间来源
   - 本次使用了哪些本地数据、哪些网络数据
   - 哪些关键数据缺失，以及如何降级
2. `大盘环境`
   - 今天市场整体强弱
   - 风险偏好、成交、风格偏向
   - 这一步结论对目标股是顺风还是逆风
3. `板块判断`
   - 目标股所属题材 / 行业 / 概念
   - 板块处于加强、分化、轮动、退潮还是超跌反抽
4. `对标股联动`
   - 默认至少列 `3` 只最相关对标股
   - 明确每只的角色：龙头 / 中军 / 高弹性 / 跟风 / 独立逻辑
   - 明确每只最新表现：涨跌、强弱、是否高于/低于目标股
   - 最后给一句联动结论：目标股在板块里处于领先、中位还是掉队
5. `目标股结构`
   - 月/周/日趋势
   - 当日分时或最新走势
   - 关键支撑、压力、强弱分界
   - 主力意图判断
6. `交易结论`
   - 是否适合建仓 / 加仓 / 持有 / 减仓 / 观望
   - 如果适合，只能写轻仓试错、分批、突破确认这类具体口径，不能空泛
   - 如果不适合，明确写原因和等待条件
7. `置信度评分`
   - `数据完整度`：0-100%（核心维度缺失越多分越低）
   - `建议行动`：立即执行 / 观察确认 / 等待数据

### 表格格式规范（2026-04-17 新增）

为提升报告可读性，以下四个部分**必须使用表格格式**输出：

**1. 最近10日走势**（目标股结构部分）

| 日期 | 收盘价 | 涨跌幅 | 换手率 | 成交量 | 形态/信号 |
|------|--------|--------|--------|--------|-----------|
| YYYY-MM-DD | X.XX | +X.XX% | X.XX% | XX万手 | 描述 |

**2. 分时主力意图分析**（四个关键时窗）

| 时间窗口 | 价格区间 | 量能表现 | 主力行为 | 信号判断 |
|----------|----------|----------|----------|----------|
| 09:30-09:35 | 价格/开盘情况 | 量能描述 | 行为描述 | 中性/偏强/偏弱 |
| 09:48-09:56 | 价格区间 | 量能描述 | 行为描述 | 中性/偏强/偏弱 |
| 11:25-11:30 | 价格/站上均价线情况 | 量能描述 | 行为描述 | 中性/偏强/偏弱 |
| 13:00-13:30 | 价格区间 | 量能描述 | 行为描述 | 中性/偏强/偏弱 |
| 14:55-15:00 | 收盘价 | 量能描述 | 行为描述 | 中性/偏强/偏弱 |

**3. 近5日资金流向**（筹码与资金部分）

| 日期 | 净流入/流出（万元） | 大单（万元） | 中单（万元） | 小单（万元） | 市场含义 |
|------|---------------------|--------------|--------------|--------------|----------|
| YYYY-MM-DD | +XXXX / -XXXX | 买入/卖出 | 买入/卖出 | 买入/卖出 | 描述 |

**4. 消息面**（消息面部分）

公司公告表格：

| 日期 | 公告内容 | 类型 | 影响判断 |
|------|----------|------|----------|
| YYYY-MM-DD | 公告标题 | 业绩/分红/重组等 | 偏利多/偏利空/中性 |

消息面综合判断：

| 维度 | 判断结果 | 说明 |
|------|----------|------|
| 消息方向 | 偏利多 / 偏利空 / 中性 / 多空交织 | 具体说明 |
| 消息级别 | 个股级 / 板块级 / 市场级 | 影响范围 |
| 消息新鲜度 | 盘中新增 / 当日新增 / 近3日 / 旧消息重炒 | 时间判断 |
| 消息可信度 | 公告实锤 / 主流媒体 / 二手转述 / 未证实传闻 | 可信度评估 |
| 情绪作用方式 | 个股直驱 / 板块带动 / 情绪扩散 / 噪音 | 作用机制 |

**表格格式要求**：
- 表格必须包含表头行
- 数据对齐方式：数字右对齐，文字左对齐
- 涨跌幅必须带正负号（+/-）
- 成交量单位统一为"万手"
- 信号判断列使用固定标签：中性/偏强/偏弱/强/弱

#### 飞书汇报特殊处理（2026-05-26 新增）

当报告通过 `send_message` 发送至飞书群聊/私聊时，**标准 Markdown 表格不会被渲染**。必须将表格转换为等宽代码块格式：

- 使用 ` ``` ` 代码块包裹，内部用空格/制表符对齐 + Unicode 制表线 `─` `│`
- 中文字符占 2 个等宽宽度，英文/数字占 1 个
- 数字右对齐，文字左对齐
- 本地保存的 `.md` 文件继续使用标准 Markdown 表格；发送飞书前动态转换

详见 `references/feishu-formatting-guide.md`。

#### 对标股数据不足时的最小合格格式

如果经过本地概念表、浏览器、API 三种渠道后，仍无法获取足够对标股数据（可用对标股 0 只），必须至少写成：

- `对标股数据缺失：本地概念成分表缺失/空，浏览器回退亦未找到可用对标股`
- `已确认的对标关系：青山纸业属于 AI硬件/CPO 题材跟风，造纸板块中军为太阳纸业，CPO 龙头为中际旭创`
- `本次联动判断可信度：低`

禁止直接省略对标股联动模块或只在正文里顺带一句。

默认至少写成下面这种曝光形式，不能只在正文里顺带提一句：

- `对标股 1：名称 代码`
  - 角色：
  - 最新表现：
  - 对目标股启示：
- `对标股 2：名称 代码`
  - 角色：
  - 最新表现：
  - 对目标股启示：
- `对标股 3：名称 代码`
  - 角色：
  - 最新表现：
  - 对目标股启示：

- `对标股数据缺失：`

### 建仓判断的固定输出

当用户问“适不适合建仓 / 能不能买 / 现在能不能上”时，结论段必须显式回答：

- `结论：适合轻仓试仓 / 适合分批建仓 / 暂不适合建仓 / 只适合观察`
- `理由：`
- `前提条件：`
- `失效条件：`

不能只说“可以考虑”“看起来还行”“不建议太重”这类模糊表述而不给明确归类。

### 持仓默认规则

- 若 `portfolio.yaml` 中该股票有持仓（`hold > 0`），自动按 `持有中` 视角输出，无需用户重复输入成本价
- 用户明确提供成本价、仓位、浮盈浮亏或"我现在持有"时，按 `持有中` 视角输出，并同步更新 `portfolio.yaml`
- 用户没有提供持仓信息且 `portfolio.yaml` 中无持仓时，默认按 `未持有` 视角分析，不要擅自切到持仓去留模式
- `持有中` 视角优先回答：
  - 继续持有 / 减仓 / 做T / 防守条件
  - 高开 / 平开 / 低开的分场景预期
- `未持有` 视角优先回答：
  - 能不能追
  - 等回踩还是等突破
  - 失效条件

#### 用户提供部分交易信息时的持仓推算陷阱（2026-05-29 新增）

当用户只提供当日的部分买卖记录（如"今天买了2100股"），但报出的盈亏率与仅含当日交易的计算结果不符时（如计算显示-3.46%但用户说+4.31%），**禁止直接假设用户计算错误**。应立即推断用户还有更早的底仓，并主动反推早期持仓的均价和数量。

**处理流程**：
1. 用用户报的盈亏率 + 当前价 + 当日交易 → 反推总股数、总成本、综合均价
2. 用反推出的总成本 - 当日交易成本 → 得到早期持仓的总成本和均价
3. 向用户确认推算结果，获得纠正后再更新 portfolio.yaml
4. **禁止**直接告诉用户"你算错了"——更可能是你缺少信息

**示例**（2026-05-29 实战）：
- 用户说"今天5.44买1000+5.16买1100，盈利4.307%"
- 仅看当日交易：均价5.293，当前5.11 → 应亏3.46%
- 但用户说盈利4.307% → 推断有早期底仓
- 反推：总3000股，综合均价≈4.90，早期900股均价≈3.98
- 确认后更新持仓

### 个人持仓数据自动加载

本技能支持自动加载并渲染个人持仓数据，用户无需每次重复输入成本价和仓位。

#### 持仓数据源

- **文件位置**：`references/portfolio/portfolio.yaml`
- **格式**：YAML，记录每只股票的成本价、持仓量、交易流水和备注
- **加载器**：`scripts/data/portfolio_loader.py`

#### 数据格式示例

```yaml
positions:
  600103.SH:
    name: "青山纸业"
    avg_cost: 3.925
    hold: 2100
    trades:
      - {date: "2026-04-10", action: "buy", quantity: 600, price: 4.360}
      - {date: "2026-04-14", action: "sell", quantity: 600, price: 4.730}
    notes: "主力持仓股"
  002347.SZ:
    name: "泰尔股份"
    avg_cost: 8.407
    hold: 0
```

#### 自动渲染规则

- `build_stock_report.py` 自动读取 `portfolio.yaml`，将持仓数据注入 JSON payload
- `render/report_renderer.py` 在报告开头自动渲染 `## 持仓情况` 区块
- 持仓区块位于 `## 一、大盘与板块环境` 之前，优先展示个人盈亏和交易记录
- 若某股票无持仓且无交易记录，则跳过持仓区块，不渲染空内容

#### 输出内容

持仓区块包含：
- 股票名称与代码
- 持仓量（股）
- 成本价（元）
- 当前价（元，基于分析日收盘价）
- 浮盈浮亏（金额 + 百分比）
- 交易记录（日期、买卖方向、数量、价格）
- 备注

#### 维护方式

- 用户通过直接编辑 `references/portfolio/portfolio.yaml` 维护持仓
- 新增交易后手动追加 trades 记录并更新 avg_cost / hold
- 清仓后将 hold 设为 0，保留历史交易记录

### 报告分层交付规则（标准版 -> 详细版）

默认先交付一份 `标准版报告`，再在结尾主动询问是否需要升级到 `详细版`，不要一开始就直接输出超长报告，除非用户明确要求详细版。

- `标准版报告` 最小内容：
  - 场景与数据
  - 大盘环境
  - 板块判断
  - 对标股联动
  - 目标股结构
  - 交易结论
- `详细版报告` 适用场景：
  - 用户明确说要详细版
  - 用户明确说要做T执行版 / 盘中执行版
  - 用户要求多场景剧本、仓位动作、风控阈值

**"做T执行版"（盘中 T+0 execution playbook）**：
用户主动索要时的标准交付物，必须包含以下固定模块：
1. `核心结论与前提` — 当日复盘定性判断（强势/弱势/分歧）
2. `关键价位与强弱分界` — 当日最低、最高、均价、主力成本区、支撑/压力
3. `三时点打分表` — 开盘初段/午后开盘/盘前最后三分钟的各维度评分
4. `多场景推演`（至少三种）— 平开强势/平开弱势/深度回调
5. `触发条件` — 买入触发、卖出触发、停手触发
6. `日内时段操作建议`（根据实时场景动态生成）
7. `仓位与风控`
8. `收盘后验证清单`

做T执行版不是另起炉灶，而是基于标准版报告的核心结论，反向拆解为具体的盘中操作指南。

详细版默认结构（供其他 agent 对齐）：

1. `核心结论与前提`
2. `关键价位与强弱分界`
3. `多场景推演`
   - 大幅高开有缺口
   - 小幅高开无缺口
   - 平开
   - 小幅低开无缺口
   - 大幅低开有缺口
4. `盘中触发条件`
   - 高抛触发
   - 低吸触发
   - 停手触发
5. `仓位与风控`
   - 单笔仓位建议
   - 当日最大回撤容忍
   - 失效后的降级动作
6. `收盘后验证清单`

### 结尾强制追问规则

标准版报告结尾默认追加下面两类追问，除非用户已明确回答过：

- `详细版追问`：
  - `要不要我基于这份标准版，继续生成做T详细执行版（含三时点打分表）？`
- `持仓信息追问`：
  - 若 `portfolio.yaml` 中无该股票数据：`你现在是否已持有这只票？如果持有，请给我成本价和大致仓位，我会切换到持仓视角重算。`
  - 若 `portfolio.yaml` 中已有该股票数据：`我已从 portfolio 中读取你的持仓信息（成本价 X.XXX 元，持仓 YYYY 股），请确认是否准确，或提供更新后的交易记录。`

执行要求：

- 若 `portfolio.yaml` 中无该股票持仓，即使默认按 `未持有` 输出，也必须在结尾主动确认用户是否持有以及成本价
- 若 `portfolio.yaml` 中已有该股票持仓，结尾主动确认持仓信息准确性
- 用户一旦补充或更新持仓成本，必须同步更新 `portfolio.yaml`
