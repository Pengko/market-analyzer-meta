# 拆单识别分析框架

> 2026-06-12 实战验证：通过东方财富 tick data 分笔成交确认主力拆单行为。
> **⚠️ 数据归属警示（2026-06-12 教训）：** 引用压缩上下文中的数据前，必须通过实际 API 调用验证其来源和数值。「同花顺暗盘/明盘」不是 API 原生字段（同花顺资金流页面仅有大单(主力)/中单/小单分类），而是通过跨源对比（东方财富超大单+大单 vs 同花顺大单(主力)）推算出的分析框架。不要将其呈现为某个具体接口的直接输出。

## 一、什么是拆单

主力将大单（超大单/大单）拆成多笔中单或小单执行，规避大单监控系统的检测。典型手法：
- 将 500万+ 的大单拆成 20-50 笔 10-20万 的中单
- 每笔控制在大单监控阈值以下（通常 <100万）
- 通过算法交易系统自动执行，时间间隔均匀

## 二、识别方法

### 方法1：笔均手数监控（最直接）

**原理**：正常交易的笔均手数有稳定区间，拆单会导致笔均异常偏低。

| 笔均手数 | 对应金额(约) | 判断 |
|----------|-------------|------|
| >50手 | >40万元 | 正常或偏大单 |
| 30-50手 | 24-40万元 | 正常区间 |
| 15-30手 | 12-24万元 | 偏小，可能存在拆单 |
| <15手 | <12万元 | 高度疑似拆单 |
| <10手 | <8万元 | 确认拆单 |

**注意**：需要排除集合竞价（09:25-09:30和14:57-15:00）的特殊情况，集合竞价笔均天然偏低。

### 方法2：中单异常放大

**原理**：如果超大单/大单缩水中单膨胀，说明大单被拆成中单。

**检测指标**：
- 中单占大+中单比 > 60% → 拆单嫌疑
- 中单占大+中单比 > 80% → 高度疑似拆单
- 中单净买入方向持续一致（>55%的分钟为净买） → 确认拆单

**与东方财富数据对比**：
```
东方财富显示: 超大单-XXX万 大单-XXX万 中单+XXXX万 小单-XXX万
解读: 大单监控看到主力在卖，但中单在买 → 拆单买入
```

### 方法3：跨源口径差异推算（最可靠的间接方法）

**原理**：东方财富将资金流分为四档（超大单>500万 / 大单100-500万 / 中单20-100万 / 小单<20万），同花顺仅分三档（大单(主力) / 中单 / 小单）。两者对「主力」的定义口径不同。当东方财富显示超大单+大单净卖而中单净买时，差额部分就是通过中单通道完成的隐蔽主力行为。

**⚠️ 重要澄清**：同花顺资金流页面（`stockpage.10jqka.com.cn/{code}/funds/`）不提供「暗盘/明盘」字段。所谓「暗盘」是通过对比两个平台的数据推算出来的分析概念，不是某个 API 的原生输出。同花顺 L2（付费版）可能有更细的订单流数据，但标准页面只有三档分类。

**检测方法**：
```
东方财富口径:
  超大单(>500万): -X亿  大单(100-500万): -Y亿
  → 明盘主力 = -(X+Y)亿（大单监控看到的）
  中单(20-100万): +Z亿
  → 若 Z 显著为正且 > |X+Y|，中单通道里可能藏着拆单买入

同花顺口径:
  大单(主力): -A亿  中单: +B亿
  → 同花顺的「大单(主力)」≈ 东方财富的「超大单+大单」
  → 若同花顺大单净卖但中单净买，与东方财富数据交叉一致 → 确认拆单
```

**推算公式**：
```
拆单量 ≈ 中单净买 - 中单历史均值（或 0）
实际主力净流入 ≈ 明盘主力(超大单+大单) + 拆单量
```

### 方法4：成交笔数异常

**原理**：同样的金额，拆单后笔数会显著增加。

**检测指标**：
- 对比历史同金额时段的笔数
- 笔数比历史均值高 30%+ → 拆单嫌疑
- 笔数比历史均值高 50%+ → 高度疑似拆单

## 三、数据获取

### 东方财富分笔成交数据（tick data）

```
GET http://push2.eastmoney.com/api/qt/stock/details/get
    ?secid={mkt}.{code}
    &fields1=f1,f2,f3,f4
    &fields2=f51,f52,f53,f54,f55
    &pos={page}
```

**返回格式**（逗号分隔）：
```
时间,价格,手数,笔数,方向
"14:56:38,7.91,222,19,1"
```

- 手数：该时间点的总成交量（手）
- 笔数：该时间点的交易笔数
- 方向：1=买入，2=卖出
- pos参数：分页，pos=0为最新，递增获取更多历史

**⚠️ API特性（关键陷阱）：**
- **pos参数必须带！** 不带pos只返回最后10条（约30秒数据），无法获取全天分笔。必须带`pos=0`才能获取全天数据（太极实业实测：4944条记录）。
- 盘后可正常访问（不像fflow/kline接口盘后返回空）
- 每页返回约20-50条记录
- 需要遍历多页获取全天数据
- 遍历时注意去重（不同pos可能返回重叠数据）

**代码模板**：
```python
import json, urllib.request

def get_tick_data(code, market, max_pages=50):
    """获取全天分笔成交数据"""
    all_details = []
    for i in range(max_pages):
        url = (f"http://push2.eastmoney.com/api/qt/stock/details/get?"
               f"secid={market}.{code}&fields1=f1,f2,f3,f4"
               f"&fields2=f51,f52,f53,f54,f55&pos={i}")
        req = urllib.request.Request(url, headers={
            'User-Agent': 'Mozilla/5.0',
            'Referer': 'http://quote.eastmoney.com'
        })
        try:
            with urllib.request.urlopen(req, timeout=3) as resp:
                data = json.loads(resp.read())
                details = data.get('data', {}).get('details', [])
                for d in details:
                    if d not in all_details:
                        all_details.append(d)
                if len(details) < 3:
                    break
        except:
            break
    
    parsed = []
    for d in all_details:
        parts = d.split(',')
        if len(parts) == 5:
            parsed.append({
                'time': parts[0],
                'price': float(parts[1]),
                'vol': int(parts[2]),   # 手
                'cnt': int(parts[3]),   # 笔数
                'dir': int(parts[4])    # 1=buy, 2=sell
            })
    parsed.sort(key=lambda x: x['time'])
    return parsed
```

## 四、分析流程

### 步骤1：获取分笔成交数据

使用东方财富 tick data API 获取全天分笔成交。

### 步骤2：计算笔均手数

```python
total_vol = sum(r['vol'] for r in records)
total_cnt = sum(r['cnt'] for r in records)
avg = total_vol / total_cnt  # 笔均手数
```

### 步骤3：分时段检测

按5分钟或15分钟分组，计算每个时段的笔均手数，找出笔均异常偏低的时段。

```python
# 按5分钟分组
five_min_groups = {}
for r in records:
    h, m = int(r['time'].split(':')[0]), int(r['time'].split(':')[1])
    key = f"{h:02d}:{(m//5)*5:02d}"
    if key not in five_min_groups:
        five_min_groups[key] = []
    five_min_groups[key].append(r)

# 检测异常时段
for key, recs in sorted(five_min_groups.items()):
    total_v = sum(r['vol'] for r in recs)
    total_c = sum(r['cnt'] for r in recs)
    avg = total_v / total_c if total_c > 0 else 0
    if avg < 15 and total_v > 1000:
        print(f"⚠️ {key}: 笔均{avg:.1f}手, 总{total_v}手 → 疑似拆单")
```

### 步骤4：中单占比分析

结合东方财富资金流数据，计算中单占大+中单的比例。

### 步骤5：跨源口径差异验证

对比东方财富四档数据与同花顺三档数据，计算中单通道中的隐蔽主力行为。注意：不要把跨源对比的推算结果呈现为某个 API 的直接输出。

### 步骤6：综合判断

| 信号组合 | 判断 |
|---------|------|
| 笔均<15手 + 中单占比>60% | 拆单嫌疑 |
| 笔均<10手 + 中单占比>80% | 高度疑似拆单 |
| 东方财富超大单+大单卖 vs 中单买 + 笔均低 | 确认拆单（跨源验证） |
| 笔均低但中单方向持续一致 | 确认拆单方向 |

## 五、拆单方向判断

拆单本身不决定方向，需要结合其他指标：

| 场景 | 笔均 | 中单方向 | 小单方向 | 判断 |
|------|------|---------|---------|------|
| 拆单买入 | 低 | 净买 | 净卖 | 主力拆单吸筹，散户抛售 |
| 拆单卖出 | 低 | 净卖 | 净买 | 主力拆单出货，散户接盘 |
| 双向拆单 | 低 | 均衡 | 均衡 | 主力对倒或换仓 |

## 六、与资金流分析的联动

拆单识别是资金流分析的高级扩展：

1. **补充明盘盲区**：东方财富明盘只统计超大单和大单，拆单后的中单不计入主力
2. **修正主力判断**：如果发现拆单，需要将中单中的拆单部分重新归类为"主力行为"
3. **交叉验证**：拆单识别结果应与同花顺数据（大单(主力)vs中单）做跨源口径一致性验证

**修正后的主力净流入**：
```
实际主力 = 明盘主力(超大单+大单) + 拆单中单
拆单中单 ≈ 东方财富中单净买 - 中单历史均值
交叉验证: 同花顺大单(主力)净卖 + 中单净买 → 与东方财富口径一致则确认
```

## 七、常见陷阱

1. **集合竞价干扰**：09:25-09:30和14:57-15:00的笔均天然偏低，不属于拆单
2. **小盘股自然笔均低**：小盘股流动性差，笔均手数本身就低，需要对比历史均值
3. **算法交易混淆**：正规算法交易（如TWAP/VWAP）也会产生均匀小单，但方向不一定一致
4. **日内波动**：盘中某些时段自然笔均偏低（如11:00-11:30交投清淡期），需排除
5. **API数据延迟**：tick data可能有几秒延迟，极端行情下不完全准确
6. **⚠️ 压缩上下文数据引用陷阱（2026-06-12 教训）**：上下文压缩后，之前的分析数据可能丢失来源信息。引用压缩前的任何数据（如「同花顺暗盘+16100万」）时，必须先通过实际 API 调用验证其来源和数值。如果无法验证，应标注「来源待核实」或直接重新获取数据，绝不能将未验证的数据呈现为某接口的直接输出。用户会追问数据来源（如「你是通过同花顺哪个接口查询到暗盘/明盘差值？」），无法回答将严重损害分析可信度。
